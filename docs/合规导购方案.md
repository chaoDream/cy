# 合规导购方案（避免比价红线）

> 目标：在不违反京东联盟 / 多多进宝任何明文规则的前提下，让「分享链 → 购买」尽量
> 不落入**比价订单**判定（比价订单佣金为 0 / 0.01 元），从而真正拿到 CPS 返佣。
>
> 本方案与既有合规红线（见 `README.md` / `docs/真到手价.md` / `data-notice`）完全兼容：
> 仅用官方 API、不爬不抓 Cookie、不承诺最低价、AI 不编价、返佣透明披露。

---

## 1. 核心认知：红线在哪

平台的「比价订单」判定，针对的是这条路径：

> 用户先在平台主站（App/小程序/H5）浏览过商品详情 → **短时间内**通过推广链接购买**同一商品**。

结论：

- **红线不在「用户粘贴分享链」这个动作**，而在「系统拿原同款 SKU 去回链购买」这个结果。
- 「合规」≠「有佣金」。用官方 API 查价不违规，但**同款回链成交**很可能被判比价、佣金归零（拼多多尤其严格）。

---

## 2. 核心策略（一句话）

> **一个输入框承接所有用户（粘贴分享链或搜索机型都行）→ 后端统一降维成关键词 →
> 差异化 SKU + 跨平台推荐购买 → 同款只展示不主推（拼多多同款先过官方比价预判）。**

不拒绝任何用户，从入口结构上消除「同款精准回链」这一比价路径。

用户视角示例（你查的是京东 iPhone 17 黑色）：

```
你查的这款：京东 iPhone 17 黑色 256G  ¥X        ← 仅展示，建立信任
更优选择（主购买入口）：
  · 京东 iPhone 17 白色 256G  ¥X-Y               ← 不同 SKU
  · 拼多多百亿补贴 iPhone 17  ¥X-Z               ← 跨平台增量新客
  · iPhone 16 / 配套手机壳膜                      ← 差异化 + 画像分散
```

## 3. 分层落地（映射到现有代码）

### 第 1 层 · 输入脱敏 —— `JdLinkParser` / `PddLinkParser`

- 新增 `extractKeyword(linkText)`：剔除颜色 / 容量 / 版本 / 自营等规格词。
  例：`iPhone 17 256G 黑色 自营` → `iPhone 17`。
- 精准 SKU/goodsSign 仍解析，但**只用于展示「你查的这款的价格」**，不作为主购买回链。

### 第 2 层 · 统一输入与解析汇流 —— `search` 页 + `AnalysisService`

- 前端：搜索框与粘贴链合并为**一个智能输入框**（保留剪贴板一键查）。
- 后端：`parseLink` 与 `search` 汇流到同一条「关键词召回」管线：
  1. 含链接 → 解析原同款，仅用于展示 + 参考价。
  2. 不含链接 → 直接当关键词。
  3. 两条路都 `extractKeyword()` → 共用差异化/跨平台推荐引擎。

### 第 3 层 · 差异化推荐引擎 —— `AnalysisService.crossPlatform`

- 标品（手机）：优先推不同颜色 / 容量 / 代际（白色、256G、iPhone 16）。
- 跨平台增量：京东链优先推拼多多百亿补贴同款，反之亦然 —— 对接收平台是新客，红线对你失效。
- 同规格同平台占比压低、后置（同规格占比 < 25%，排第 4 位后）。
- 排序默认综合 / 销量，不默认价格升序。
- 可混入配套品（壳 / 膜）丰富推荐、分散流量画像。

### 第 4 层 · 拼多多官方预判 + 备案 —— `PddDdkClient` / `PddDdkService`（必做，原本缺失）

- 备案：`pdd.ddk.rp.prom.url.generate`（带 `custom_parameters`，如 `{"uid":"<用户ID>"}`）。
- 查备案：`pdd.ddk.member.authority.query`。
- 搜索 / 转链统一带 `pid` + `custom_parameters`，读取返回的 `predict_promotion_rate`。
- `buildCpsLink` 前置判断：
  - `predict_promotion_rate == 0` → 不出购买按钮，转纯展示 / 引导搜同款 / 切京东。
  - `predict_promotion_rate > 0` → 正常生成 PID 链接。

### 第 5 层 · 合规跳转 + PID 管理 —— `AffiliateProperties` / 跳转

- 走自己已备案的小程序 / H5 来源跳转，PID / 来源齐全（已有 `positionId`）。
- PID：在**同一个合规账号**下按品类 / 入口开 2~5 个推广位轮询，便于数据区分与负载分散。
- 保留 `analysis.js` 已有的返佣披露弹窗。

### 第 6 层 · 数据验证 —— `Track`

- `purchase_click` 增加 `is_price_compare` 维度（来自 PDD 预判结果）。
- 统计各入口（搜索 vs 粘贴链）的比价率、差异化推荐 vs 同款的成交占比。

## 5. 明确不做（企业主体的两条硬红线）

1. ❌ 用他人身份注册多个联盟账号组矩阵（违反「一主体一账号」协议，连带封号）。
   - 注意：**同一账号下开多个推广位 PID 是允许的**，这才是「PID 轮询」的正确做法。
2. ❌ 伪造你不拥有的 Referer / 来源标识（经自己备案页中转 ✅，捏造来源 ❌）。

其余手法（前端延迟、本地缓存、跨平台、差异化推荐、裂变分享、同账号多 PID、备案中转）均在规则内。

---

## 6. 为什么这套最适合本项目

- 不推翻任何既有红线，只在 `data-notice` 合规叙事上自然延伸。
- 复用现有 `crossPlatform` / `priceEngine` / `Track` / `positionId` / Redis 快照。
- 只补一个真缺口：拼多多官方预判 + 备案。
- 核心不是对抗算法，而是改流量结构——让佣金来自平台真正愿付费的「差异化 + 跨平台增量」。
